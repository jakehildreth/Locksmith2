# ADR-0004: GitHub Release Publishing Strategy for Locksmith2 Build

## Status

Accepted

## Context

Locksmith2 is built with [PSPublishModule](https://github.com/EvotecIT/PSPublishModule) v2.0.27. The current build process in [Build/Build-Module.ps1](../Build/Build-Module.ps1) produces both unpacked and packed artefacts via `New-ConfigurationArtefact`.

PSGallery publishing is intentionally performed outside the `Build-Module {}` DSL in [Build/Invoke-LS2PostBuildPublish.ps1](../Build/Invoke-LS2PostBuildPublish.ps1). After the build completes, that script vendors `PSWriteHTML` and `PSCertutil` into the unpacked artefact and then calls `Publish-Module -Path`. This avoids a known issue where PSPublishModule's in-DSL publish path uses `Publish-Module -Name`, which resolves against `PSModulePath` and publishes the pre-vendoring copy of the module, excluding the vendored dependencies.

We want to add GitHub Release publishing to the build so that packed release assets are automatically attached to versioned GitHub releases. The release asset must contain the same vendored `PSWriteHTML` and `PSCertutil` dependencies that ship to PSGallery.

## Decision

1. Keep all publishing logic inside [Build/Invoke-LS2PostBuildPublish.ps1](../Build/Invoke-LS2PostBuildPublish.ps1). PSGallery publishing remains `Publish-Module -Path` after vendored dependencies are injected into the unpacked artefact.
2. GitHub Release publishing is added to the same post-build function as an independent operation. After vendoring, the vendored unpacked artefact is compressed into a zip and uploaded to a GitHub release via the GitHub REST API. GitHub release creation does not depend on PSGallery publishing being requested.
3. GitHub Release publishing is controlled by a dedicated `-PublishToGitHub` switch and requires a GitHub token (`-GitHubAPIKey` or `-GitHubAPIPath`). The workflow passes `-PublishToGitHub` in both release and prerelease build steps.
4. GitHub releases are CI-only. `Invoke-LS2PostBuildPublish` requires a `target_commitish` value, supplied via the `-GitHubSha` parameter or inherited from `$env:GITHUB_SHA`. When `-GitHubSha` is empty and `$env:GITHUB_SHA` is not set, the function fails loudly. This prevents local developer runs from creating releases tied to the default branch.
5. The packed zip asset is created on demand from the vendored unpacked artefact; it is not the `New-ConfigurationArtefact -Type Packed` output from PSPublishModule. The artefact is staged under a temporary `Locksmith2/` parent folder before compression so the extracted archive matches PowerShell's module folder convention and can be dropped directly into a `Modules/` directory.
6. GitHub release metadata follows branch semantics:
   - `main` branch builds produce stable releases.
   - Non-`main` branch builds produce prereleases.
7. Default GitHub coordinates are owner `jakehildreth` and repository `Locksmith2`; both can be overridden via build parameters.
8. The release tag is derived from the module version plus any prerelease suffix.
9. The release target commitish is derived from `-GitHubSha` (parameter) or `$env:GITHUB_SHA` (environment), defaulting to the environment value for GitHub Actions compatibility.

## Consequences

[+] GitHub releases are created automatically alongside PSGallery releases using the same trigger.
[+] The GitHub release asset contains the exact same vendored dependencies as the PSGallery publish because both use the same post-build artefact.
[+] PSGallery and GitHub release publishing can be requested independently via their own switches.
[+] No changes are required to the `Build-Module {}` DSL or to artefact configuration in PSPublishModule.
[+] Branch-based prerelease semantics keep stable and prerelease releases distinct.
[!] `Invoke-LS2PostBuildPublish` grows in responsibility and must be maintained as the single publish orchestrator.
[!] GitHub release creation requires a token with appropriate repository permissions.
[!] The GitHub release asset name and release notes are hand-rolled rather than generated by PSPublishModule.

## Test Impact

- [Tests/Build/Build-Module.Tests.ps1](../Tests/Build/Build-Module.Tests.ps1) should be extended to assert that `Invoke-LS2PostBuildPublish` creates a GitHub release when a token is supplied and publishing is enabled.
- Tests should verify the zip asset is created from the vendored artefact and uploaded via `Invoke-RestMethod`.
- Branch-based prerelease behavior should be covered by tests that inspect the release payload.
- Existing PSGallery publishing tests should continue to pass; they may need mocks for `Invoke-RestMethod` and `Compress-Archive`.
